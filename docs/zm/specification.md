一、需求简述
（一）需求背景
当前主流 Git 图形化工具（如 Sourcetree、GitHub Desktop 等）主要停留在对 Git 命令的图形封装层，缺乏对代码语义的理解能力，导致：
Commit 信息质量低，难以追踪变更意图
冲突发现滞后，仅在操作后暴露问题
Git 操作门槛高，新手易误操作
随着 AI 与代码分析技术的发展，有必要构建一款具备语义理解能力 + 智能辅助能力的 Git 工具，提升开发效率与代码质量。
（二）价值评估
本项目的核心价值体现在：
提升团队 Commit 质量与规范性
提前发现冲突风险，降低合并成本
降低 Git 学习门槛，提高开发效率
利用 AI 提供自动化辅助与决策支持
最终目标是打造一个**“从操作工具 → 智能开发助手”转变的 Git 客户端**。
（三）产品简介
产品名称
IntelliGit 智能 Git 桌面客户端
产品定位
面向开发者的 AI 驱动 Git 管理工具
产品特色
语义级代码理解（AST 分析）
AI 自动生成 Commit
冲突预检测（操作前提示）
本地知识库增强（RAG）
自然语言操作 Git
目标用户
有基础 Git 使用经验的开发者
中小型开发团队

（二）系统架构说明
IntelliGit 采用「微内核 + Sidecar」的混合进程架构，将 Git 引擎与 UI 进程解耦，保证高性能 Git 操作的同时维持前端开发体验的灵活性。

三层进程模型：
•Renderer 进程（React/TypeScript）：负责 UI 渲染与用户交互。
•Main 进程（Electron/Node.js）：核心调度层，负责 Go Sidecar 生命周期管理、引擎路由决策、Git CLI 调用。
•Go Sidecar 进程：高性能 Git 操作引擎，基于 libgit2 实现，通过 JSON-RPC 2.0 over stdin/stdout 与 Main 进程全双工通信。
特点：
高性能 Git 操作
模块解耦
支持 AI 扩展能力

顶部全局工具栏：
内容从左到右：
应用 Logo / 名称「IntelliGit」（点击返回概览页）

自然语言输入框

右侧：
- 快捷操作按钮组：Fetch、Pull、Push（图标 + 文字，Push 右侧附带角标显示待推送 commit 数量）
- 设置齿轮图标
Fetch/Pull/Push 的状态反馈：按钮点击后变为 loading 状态，右上角出现进度 toast，完成后恢复并刷新状态。

左侧导航栏：	
自上而下：

| AI 助手 | NLP 命令中心 |

| 设置 | 设置页 |
底部状态栏：
左侧：当前分支名 → 点击同顶部分支选择器
中间左：↑3 ↓1（与远程的差异，待推送/待拉取数量）

（二）核心页面设计
1：commit页面

整体布局：四栏结构
页面由左至右分为四个区域，各司其职：
左侧导航栏（最窄） 是全局入口，以图标形式承载提交、分支、警告、历史、设置等模块切换。底部两个状态点实时反映「引擎就绪」与「API 已连接」两个系统状态。当前页高亮「提交」图标，并以红色数字徽章标示未处理变更数量。
文件列表区 是工作流的起点。变更被分为「已暂存」与「未暂存」两个可折叠区域，每条文件项展示状态标识（M / A / D）、文件名、行数增减，支持单文件右键操作（暂存、忽略、历史等）。顶部的「按意图分组」开关开启后，AI 会将所有变更重新聚合为语义意图组（如"feat: 新增认证逻辑"、"refactor: 清理旧工具函数"），允许用户按意图分批提交，是核心差异化功能。
Diff 视图区 是信息密度最高的区域。点击左侧文件即切换对应 diff。每个 Hunk 块头部显示 AST 语义标签（如「在 UserModel.validateToken 内」），帮助用户快速理解变更上下文而无需翻阅代码。Hunk 右侧提供「暂存此块」按钮，支持精细到块级别的暂存，不必整文件操作。顶部可切换统一视图与分栏对比视图。
提交面板（最右） 是工作流终点，承载 commit message 的撰写与提交动作。分为主题（单行，含 50 字符计数警告）和详情（多行）两个输入区，「AI 生成」按钮触发后会分析当前暂存内容，自动填入主题与详情，并以淡色提示「AI 生成，请确认后提交」引导用户校对。选项区可勾选沙箱验证。最底部主操作按钮「提交」显示已暂存文件数，次要操作「提交并推送」合并两步为一步。

核心操作流程
选文件 → 查看 Diff → 暂存（文件级 or Hunk 级）→ AI 生成提交信息 → 确认 → 提交（可选推送）
「语义化暂存」流程则在「按意图分组」开启后分叉为：
开启分组 → AI 分析所有变更 → 按意图组展示 → 选择意图组全部暂存 → 各组独立提交
这两条路径均在同一页面内完成，无需跳转，是整个工作台设计的核心思路。

2：历史和分支页面
整体布局：四栏结构（与提交页复用同一框架）
左侧导航栏 与提交页完全一致，「分支拓扑」图标高亮为当前页，底部状态指示灯保持常驻。
分支侧边栏 将本地分支与远程分支分组展示。每条分支右侧以 ↑N↓M 直观呈现与当前分支（main）的提前/滞后 commit 数量。legacy/v1-compat 因长期不活跃，名称以灰色弱化显示并标注时间距离。当前分支 main 以星号标示。右键分支条目可触发切换、合并、Rebase、删除（二次确认保护）等操作。
拓扑图画布（主体） 是页面的视觉核心，以 Canvas/SVG 渲染多分支提交历史。设计要点如下：
不同分支以色彩编码区分——蓝色为 main，青色为 feature-auth，红色为 fix/token-bug，琥珀色为 refactor/cleanup。长期不活跃的分支以虚线降调渲染。Merge Commit 节点使用菱形而非圆点，以形状区分语义。每个分支末端节点带有风险光晕，绿色表示可安全合并，红色表示存在文本冲突，琥珀色表示潜在逻辑碰撞风险——这是 AI 预检的核心输出结果。顶部工具栏提供按作者/关键词搜索和按时间段筛选，以及显示/隐藏远程分支的开关。
右侧 Commit 详情面板 在点击节点后填充具体信息：完整 commit message、作者头像缩写 + 姓名 + 精确时间戳，以及变更文件列表（以 M/A/D 标识类型）。面板底部提供三个操作入口，Reset 以红色文字标记为高危操作，视觉上与普通操作形成区分。

核心操作流程
查看拓扑图 → 识别风险分支（看光晕颜色）→ 点击节点看详情 → 右键分支发起合并预检 → 确认后 Merge / Cherry-pick / Reset
风险预检的分支状态（绿/红/琥珀）由后台影子合并分析实时计算，用户在决策前即可感知合并代价，是该页面相较于传统 Git 客户端的核心差异点。
页面 3：风险管控 / 冲突解决


布局为左右两栏，由顶部的分支选择器统领整体语境——左侧显示目标分支（main），右侧显示来源分支（fix/token-bug），可实时切换触发重新检测。
左栏是冲突文件导航与三栏 Diff 对比区。顶部红色告警横幅汇总风险总量。两个 Tab 分别承载「影子合并预检」（文本层冲突）与「语义冲突告警」（AST 语义层碰撞）。每个冲突文件展开后以 ancestor / ours / theirs 三栏并排呈现具体冲突区域，底部「AI 修复建议」按钮将分析结果推送至右侧面板。底部操作栏在所有冲突未解决时禁用「立即合并」，以状态驱动引导用户完成流程。
右栏是 AI 冲突修复助手，包含推荐策略（文字说明保留哪方逻辑及理由）、合并后的建议代码块，以及「应用建议」主操作按钮。底部追问输入框支持与 AI 进一步对话（如「为什么不保留 theirs？」），将冲突解决闭环在同一界面内完成。

页面 4：NLP 命令中心

全页面布局，核心交互围绕一个大输入框展开。用户用自然语言描述意图，AI 实时在输入框下方预览转译的 Git 命令，并标注风险等级（安全 / 高危 / 极高危）。
高危命令附带橙色说明区块，阐述操作影响（如 reset --soft 会保留工作区代码）。极高危命令（如 push --force）在历史记录中显示为「已阻止」，需在设置中显式解锁，形成二次防护。
「常用操作」快捷标签提供高频场景一键触发。页面下半部分的历史执行记录按时间倒序展示每条操作的原始语言输入、转译命令，以及成功 / 阻止的执行结果，形成完整的操作审计轨迹。

页面 5：沙箱验证

顶部绿色横幅是 Docker 环境状态的即时反馈，可用时显示版本号，不可用时引导安装，作为整个页面功能是否可用的前提声明。
配置区采用左右分区：左侧以键值对展示当前沙箱配置（镜像、工作目录、命令、超时时长），右侧提供触发时机开关（提交前 / 推送前）和手动触发按钮，配置与触发动作在同一视图完成。
验证结果区分为两层：上层是四格统计卡片（总测试 / 通过 / 失败 / 用时），以颜色编码一眼传达健康状态；下层是终端风格的完整日志输出，通过行色区分 PASS（绿）/ FAIL（红）/ 系统提示（灰），并精确展示失败断言的 Expected / Received，方便开发者直接定位问题。状态栏在沙箱运行时额外亮起「沙箱已启用」指示点，与引擎状态并列显示。
