# 项目功能优化分析报告

该项目目前已具备基本的 Git 操作能力（分支、提交、历史、远程同步、暂存等），UI 框架基于 Ant Design，侧边栏采用自定义样式。然而，在交互完整性、状态同步、布局灵活性、精细 Git 功能方面仍有较大提升空间。

---

## 1. 仓库选择控件分布不合理（存在重复）

**现状**：  
- 顶部工具栏（`Toolbar` 组件）中存在一个仓库选择下拉框（通过 `Select` 或 `Dropdown` 切换 `currentRepo`）。  
- 左侧边栏（`RepoSidebar`）中也展示仓库列表，点击可切换仓库。

**问题**：两处功能完全重叠，且布局分散，用户不易理解各自意图。

**根因**：  
设计初期未统一“仓库切换入口”的定位。工具栏本应承载全局操作，而仓库选择更适合放在左侧导航区域。

**优化建议**：  
- **二选一保留**：移除顶部工具栏中的仓库切换下拉框，保留左侧仓库侧边栏作为唯一切换入口。  
- 或者，如果希望保留顶部快速切换，可将左侧改为“仓库总览/管理”（如添加、排序、删除），不承担切换功能。  
- 参考 GitHub Desktop 设计：仓库切换放在左侧上方，顶部只显示当前仓库名称和操作按钮。

---

## 2. 缺少移除被管理仓库的功能

**现状**：  
`useAppStore` 中已实现了 `addRepo`、`createRepo`、`cloneRepo`、`switchRepo`，但**未提供 `removeRepo` 方法**。实际上，从 `useAppStore.ts` 的代码可见 `removeRepo` 已有定义（接口中），但尚未在 UI 中提供触发入口。

**影响**：用户无法清理已废弃或过时的仓库，列表会无限膨胀。

**优化建议**：  
- 在 `RepoSidebar` 每个仓库项的悬浮区域或上下文菜单中添加“移除仓库”按钮。  
- 执行前弹出确认对话框（`Modal.confirm`），确认后调用 `removeRepo(path)`。  
- `removeRepo` 内部需：从 `repos` 数组中移除、从持久化配置中删除、若移除的是当前仓库则清空 `currentRepo` 并重置所有 git 状态。

---

## 3. 仓库数量增加时，左侧功能按钮被向下挤压

**现状**：  
左侧边栏（`ig-repo-dock`）中，所有仓库缩略图垂直排列，下方没有固定区域放置其他功能按钮（例如切换视图、设置等）。实际上视图切换按钮位于侧边栏之外的“活动导轨”（`ig-activity-rail`）中，但仓库列表区域过长时会撑满整个左侧栏。

**影响**：当用户添加10+仓库时，仓库列表会把其他导航按钮挤到可视区域之外，用户需要滚动才能找到“历史”、“设置”等按钮。

**优化建议**：  
- 将左侧栏分为两个区域：  
  - 上部：仓库列表（`overflow-y: auto`，可滚动）。  
  - 下部：固定高度的功能快捷键区域（如视图切换、分支操作等）。  
- 或者，让仓库列表采用“仅显示首字母图标”的紧凑模式，水平排列（类似 VSCode），右边显示功能按钮。  
- 参考当前代码：`ig-activity-rail` 中已经有 `VIEW_OPTIONS` 和 `RepoSidebar` 的布局，可进一步优化侧边栏内部的滚动与固定关系。

---

## 4. 仓库设置界面存在意义不明的右侧边栏

**现状**：  
从 `MainApp.tsx` 看，`SettingsView` 组件渲染在 `main` 区域内部，旁边并未显式添加右侧边栏。但用户在描述中提到“设置界面存在右侧边栏”。可能指的是在历史视图（`HistoryView`）中右侧有一个用于分支筛选的边栏（见问题7）。或者有些布局下设置页也被包裹在具有双栏的容器中。

**影响**：右侧边栏无实际内容，造成视觉干扰和空间浪费。

**优化建议**：  
- 检查 `MainApp.tsx` 中是否存在全局的双列布局。如果设置视图不应有右侧边栏，则条件渲染（只在 history 视图显示右侧分支侧边栏）。  
- 对 `SettingsView` 仅保留居中表单，移除所有无关的侧边栏。

---

## 5. 部分图标表意不明

**现状**：  
`MainApp.tsx` 中使用了 Ant Design 图标（`CodeOutlined`、`HistoryOutlined`、`SettingOutlined` 等），部分图标（如 `CloudUploadOutlined`、`BranchesOutlined`）缺乏文字标签或 `title` 属性。

**影响**：新用户难以快速理解按钮功能，影响操作效率。

**优化建议**：  
- 为所有 `Button` 或 `Icon` 组件添加 `title` 属性（例如 `title="推送"`）。  
- 在 `VIEW_OPTIONS` 中使用 `label` 字段在侧边栏显示文字（当前代码已显示文字）。工具栏中的图标可以考虑加 Tooltip。  
- 对易混淆图标（如 `CloudDownloadOutlined` 用于拉取，`CloudUploadOutlined` 用于推送）保持统一，并确保文字提示一致。

---

## 6. 仓库历史界面查看 commit 内容问题，且缺少查看具体变更内容功能

**现状**：  
- `HistoryView` 已经调用 `selectCommit` 并获取 `selectedCommit` 与 `selectedCommitFiles`。  
- 但在 UI 中，点击 commit 后可能只显示了简短信息，没有“查看文件变更”面板。  
- 当前代码中 `selectedCommitFiles` 数据已加载，但缺少渲染 diff 的完整组件。

**影响**：用户无法审查每次提交的详细变更，无法替代命令行 `git show`。

**优化建议**：  
- 在历史视图右侧（或底部）增加一个“提交详情面板”，展示：  
  - commit hash、作者、日期、message。  
  - 变更文件列表（`selectedCommitFiles`）及每个文件的状态（M/A/D/R）。  
- 点击单个文件时，调用 `git diff <commit>^! -- <file>` 获取 diff，并在内联 diff 组件中展示。  
- 可利用已有的 `DiffView`（若存在）或复用 `ChangesView` 中的 diff 组件。

---

## 7. 仓库历史界面的“分支”右边栏意义不明

**现状**：  
`HistoryView` 中实现了一个 `branchFilter` 功能，右侧有一个可折叠的“分支”边栏（预期用于筛选显示哪些分支的 commit）。但从代码看，该边栏只提供了空的筛选输入框，并未真正关联历史过滤逻辑。

**影响**：用户期望看到分支图/分支切换列表，但右侧区域内容空洞。

**优化建议**：  
- 若右侧边栏意图用于分支选择，则增加实际的分支列表（`branches` + `remoteBranches`），点击后过滤 `allCommitHistory` 只显示该分支的 commit。  
- 或者改用顶部下拉框替代。  
- 至少提供用于筛选分支的 Tag 或 Checkbox，并实时更新历史列表。

---

## 8. 文字信息缺少“点击复制”便捷设计

**现状**：  
所有文本（如 commit hash、分支名、仓库路径）均为纯文字，没有提供一键复制功能。

**影响**：用户需要手动选择文本并 Ctrl+C，操作繁琐。

**优化建议**：  
- 对 commit hash、分支名、仓库路径等高频复制文本添加“复制”按钮（使用 `navigator.clipboard.writeText`）。  
- 采用 Ant Design 的 `Typography.Text copyable` 组件可快速实现。  
- 在设置页中对路径字段也可添加复制按钮。

---

## 9. 各界面侧边栏缺少收缩展开设计

**现状**：  
`ig-sidebar` 和 `ig-repo-dock` 均固定宽度，没有提供折叠/展开切换。用户在查看宽 diff 或历史时，希望临时隐藏侧边栏获得更多水平空间。

**影响**：不可折叠的侧边栏降低了小屏或大内容场景下的体验。

**优化建议**：  
- 为左侧导航栏和右侧分支边栏添加可折叠按钮（`collapsed` 状态）。  
- 在 `useAppStore` 中添加 `sidebarCollapsed` 状态，并保存到 localStorage。  
- 折叠时只显示图标或极窄条，悬浮时展开。

---

## 10. 仓库变更左侧边栏功能占比不合理，提交信息输入框超出界面

**现状**：  
`ChangesView` 中左侧边栏（`ig-sidebar`）显示了暂存/未暂存文件列表，占用约 240px 宽度。底部是 `CommitPanel`（提交信息输入框）。当文件列表较多或窗口高度较小时，提交输入框被挤压甚至溢出到底部之外。

**影响**：用户无法正常输入 commit message，属于功能 Bug。

**优化建议**：  
- 确保 `ChangesView` 容器使用 `flex` 布局，提交面板固定底部高度（如 ~120px），文件列表区域使用 `overflow-y: auto`。  
- 在 CSS 中设置 `.ig-commit-panel` 的 `flex-shrink: 0`，防止被压缩。  
- 考虑将提交面板放在右侧或底部栏，而不是侧边栏内。

---

## 11. 仓库变更中 commit 功能只能提交整一个文件，无法做到精细 commit

**现状**：  
当前 `ChangesView` 仅支持文件级别的 staging/unstaging（`addFile`/`removeFile`），没有行级或 hunk 级暂存功能。后台 sidecar 已经实现了 `ApplyPatch`、`StageHunk`、`UnstageHunk` 等 API（见 `sidecar/internal/git/staging_hunk.go`），但前端尚未集成。

**影响**：用户无法只暂存文件的部分改动，被迫一次性提交不相关修改。

**优化建议**：  
- 在 `ChangesView` 中展示文件的 unified diff（暂存/未暂存），每段 hunk 上方添加“暂存此区块”按钮。  
- 调用 `window.electronAPI.invokeGit('staging.stageHunk', { path, patchContent })`。  
- 已有 `fetchRawDiff` 和 `ApplyPatch` 接口，前端可以从中提取 patch 内容。  
- 参考 GitKraken 的“Stage Hunk”交互设计。

---

## 12. 部分信息显示控件存在状态不同步问题

**现状**：  
- 切换仓库时，`selectedCommit`、`selectedCommitFiles`、`selectedFilePath`、`workdirDiff` 等状态未清空，导致历史视图显示旧仓库的信息。  
- 提交 commit 后，`ChangesView` 中的文件列表未立即刷新，需要手动切换视图才更新。

**根因**：  
`useAppStore.switchRepo` 中只重置了 `currentRepo` 等顶层状态，但没有调用 `refreshAllLocal` 或重置 UI 相关状态（如 `selectedCommit`）。在 `createCommit` 成功后，也没有自动调用 `refreshStatus`。

**影响**：用户看到不一致的信息，极度损害可信度。

**优化建议**：  
- 在 `switchRepo` 的结尾显式 reset 所有 UI 状态：`set({ selectedCommit: null, selectedCommitFiles: [], selectedFilePath: null, workdirDiff: null, diffCompareResult: null })`。  
- 在 `createCommit` 成功后，自动调用 `refreshStatus` 和 `refreshHistory`。  
- 同时确保 `checkoutBranch` 后也刷新 UI 状态。

---

## 13. 缺少全局设置

**现状**：  
目前只有仓库级别的设置（`SettingsView`），没有全局设置界面。用户无法配置主题（亮/暗）、默认语言、自动刷新间隔、快捷键等。

**影响**：应用功能不完整，无法适应不同用户偏好。

**优化建议**：  
- 增加全局设置页面（路由 `/settings/global`）。  
- 设置项可包括：  
  - 主题模式（`light` / `dark`）  
  - 默认自动刷新（秒）  
  - 默认提交作者信息  
  - 界面语言（未来扩展）  
  - 侧边栏是否默认折叠  
- 全局设置持久化到 localStorage，在 `useAppStore` 中添加 `appSettings` 对象。  
- 当前的主题切换已在 `MainApp.tsx` 中通过 `themeMode` 实现，但保存在组件本地状态 `useState` 中，未持久化，也属于优化范畴。
