# 修复计划鲁棒性审查与增强授权 (2026-05-23)

## 概述
今天对 IntelliGit 性能优化修复执行计划（`docs\lxy\repair-plan.md`）进行了可行性与鲁棒性审查，发现原计划中提供的示例代码和部分指导思想在应对高并发或极端状态时显得过于简单。为了确保最终修复的稳定性，对文档进行了针对性的增强说明。

## 发现的不鲁棒环节与增强建议
在审查过程中，我们明确了以下需要在实际编码中加强防御的环节：
1. **Goroutine 安全与并发瓶颈**：原计划 `main.go` 并发改造中缺乏 `recover()` 机制，一旦发生 Panic 将导致整个进程崩溃；同时缺乏并发数限制，可能导致内存或系统句柄耗尽。建议使用 Worker Pool 或通道信号量来限制并发。
2. **锁粒度过粗导致读请求饿死**：`Repository` 引入全局 `sync.RWMutex` 后，长时间的网络操作（如 Fetch）如果在全过程占用写锁，将会导致前端高频轮询的本地 Status 读取长时间阻塞并造成 UI 假死。必须对长时间操作使用更精细的锁策略。
3. **Go 循环指针与性能隐患**：在遍历 `idx.Entries` 构造 Map 时，如果不当获取循环变量指针，在老版本 Go 中会引起全部数据错乱的问题。
4. **前端后台任务堆积**：前端改为异步触发 `refreshRemote` 后，如果不加防重入锁，多次刷新会引发大量后台 Fetch，造成 Git 目录锁（`FETCH_HEAD.lock`）严重踩踏。
5. **实例复用安全**：在打开仓库实例做缓存复用时，判断依据过少。如遇本地 `.git` 文件夹意外被删，复用缓存会导致不可控崩溃，应该增加目录有效性校验。

## 执行结果
在 `docs\lxy\repair-plan.md` 文档开头，插入了一段 **致执行 Agent 的特别授权说明**。
该说明正式向未来的执行者授权：**在保证功能效果和核心优化目标不偏离的前提下，可以自主进行代码结构的防御性编程与鲁棒性增强**，无需刻板遵守文档中不够周全的简化版代码，以实现具有工业级高可用性的最终交付。
